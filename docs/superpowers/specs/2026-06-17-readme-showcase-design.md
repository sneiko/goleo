# Спецификация: README showcase и демонстрационные примеры

Дата: 2026-06-17

## Контекст

Текущий README уже описывает базовые возможности Goleo, но визуально и
структурно он пока не выполняет роль витрины проекта. Скриншоты почти не
раскрывают поверхность API, а существующие demo-примеры слишком минимальны,
чтобы на их основе собрать понятную и привлекательную галерею.

Нужен README в стиле продуктовой документации: с сильным hero-блоком, набором
осмысленных скриншотов, более богатыми примерами и явным объяснением, что
именно можно собрать на Goleo.

## Цели

- Сделать README витриной проекта, а не только технической памяткой.
- Добавить отдельные showcase-примеры, из которых можно получать стабильные и
  содержательные скриншоты.
- Улучшить визуальную подачу встроенного frontend UI так, чтобы form/chat/output
  состояния выглядели цельно и убедительно.
- Сгенерировать воспроизводимый набор screenshot assets для README.
- Сохранить текущую архитектуру Goleo, не ломая существующие API и существующие
  demo-entry points.

## Не-цели

- Не менять transport/API endpoints и schema contract.
- Не переписывать frontend с нуля.
- Не заменять существующие `simple`, `chat`, `http-wrapper`, `openai-stream`,
  `ollama`.
- Не делать README зависимым от реально поднятых внешних сервисов при генерации
  скриншотов.

## Выбранный подход

Выбран умеренный вариант `showcase + UI polish`.

Суть подхода:

- Существующие интеграционные примеры сохраняются как рабочие entry points.
- Для README добавляются новые showcase-примеры, которые выглядят богаче и
  лучше раскрывают API.
- Встроенный frontend получает точечную UI-полировку, достаточную для сильных
  скриншотов и более зрелого product presentation, но без изменения базовой
  модели приложения.

Причина выбора:

- Вариант только с новыми скриншотами не даст понятной документации, потому что
  текущие примеры слишком пустые.
- Полная продуктовая переработка UI сейчас избыточна и размоет задачу.
- Комбинация richer demos и ограниченной UI-полировки дает лучший баланс между
  качеством README и риском изменений.

## Новые примеры

### `examples/showcase-form`

Назначение:

- Основной hero-демо для README.
- Источник для скриншотов form components и outputs.

Содержимое:

- `Textbox`
- `Number`
- `Slider`
- `Checkbox`
- `Dropdown`
- `File`
- Несколько outputs: человекочитаемый текст и `JSON`

Сценарий:

Пользователь заполняет форму, похожую на небольшой AI content/prompt tool.
Handler возвращает:

- короткий summary;
- структурированный JSON с echo-данными и производными значениями.

Требования:

- Осмысленные `placeholder` и `default` значения.
- Состояние по умолчанию должно быть достаточно богатым для hero-screenshot.
- Результат должен выглядеть содержательно даже в статичном кадре.

### `examples/showcase-chat`

Назначение:

- Источник для chat screenshot.
- Демонстрация стриминга в более реалистичном формате.

Содержимое:

- `Chat`
- Стриминговый ответ по частям

Сценарий:

Небольшой copilot-style assistant. На вход получает короткое сообщение и
отвечает несколькими потоковыми чанками, чтобы transcript выглядел как реальная
беседа, а не как эхо-демо.

Требования:

- Ответы должны быть краткими, но визуально правдоподобными.
- Состояние должно хорошо смотреться как в пустом, так и в заполненном виде.

### `examples/showcase-adapters`

Назначение:

- Источник для adapter-oriented screenshot.
- Пояснение позиционирования Goleo как UI-обертки над внешними backends.

Содержимое:

- Локально детерминированный demo-flow без обязательной зависимости от внешних
  сервисов.

Сценарий:

Экран, который объясняет идею "prompt in -> external backend profile ->
normalized result out" через локальный handler и осмысленные output-блоки.

Причина такого решения:

- Для README нужен воспроизводимый экран.
- Реальные `http-wrapper`, `openai-stream`, `ollama` остаются в README как
  integration commands, но не используются как обязательная основа для capture.

## Изменения во frontend UI

Изменения должны быть точечными и не менять публичный API библиотеки.

### Header и верхняя композиция

- Усилить header.
- Добавить короткий контекстный subtitle/metadata блок.
- Сделать так, чтобы страница сразу выглядела как demo surface, а не как пустой
  лист с одной карточкой.

### Form layout

- Улучшить раскладку inputs и outputs.
- На заполненных showcase-сценариях пользователь должен видеть в одном кадре и
  форму, и результат.
- Несколько interfaces на странице должны визуально читаться как curated demo
  gallery, а не как одинаковые карточки в списке.

### Output presentation

- Сделать text/json output-блоки более выразительными.
- Состояние с результатом должно выглядеть как артефакт работы инструмента, а
  не как сырой `pre`.

### Chat transcript

- Улучшить пустое состояние.
- Сделать различие user/assistant более читаемым.
- Streaming state должен выглядеть живо и понятно.

### Компоненты формы

- `File`, `Slider`, `Dropdown`, `Checkbox` должны выглядеть как осознанный
  showcase API-поверхности.
- Это не требует новых компонентных типов, только лучшей подачи существующих.

## Воспроизводимые screenshot states

Скриншоты не должны зависеть от ручного кликанья в браузере.

Для этого:

- В showcase-примерах задаются наполненные `default` значения.
- Для сложных состояний допускается отдельный demo-state механизм, например
  query-param вроде `?demo=readme`, если он ограничен showcase-сценариями.
- README assets должны пересниматься одной maintainers-only командой, а не
  серией ручных действий.

## Изменения в README

README перестраивается в более product-oriented структуру.

Порядок блоков:

1. Hero screenshot
2. Короткий pitch
3. `What you can build` с тремя screenshot blocks
4. `Built-in components` gallery
5. `First demo`
6. `Examples` с явным mapping на команды
7. `Lifecycle`, `Logging`, `Frontend development`, `Architecture`

Принципы:

- README должен сначала показывать результат, затем объяснять API.
- Скриншоты должны не дублировать друг друга, а раскрывать разные сценарии.
- Команды запуска для интеграционных примеров остаются явными и рабочими.

## Проверка

Обязательная проверка после реализации:

- `make check`
- Проверка frontend tests на richer rendering states
- Проверка, что screenshot-файлы существуют и не пустые
- Проверка, что старые примеры запускаются как раньше
- Проверка, что README не ссылается на несуществующие команды или assets

## Порядок реализации

1. Добавить showcase examples и команды запуска.
2. Доработать frontend-представление под richer demo states.
3. Сгенерировать screenshot assets.
4. Пересобрать README под новую gallery-структуру.
5. Прогнать `make check` и проверить итоговые assets.

## Риски и ограничения

- Слишком агрессивная UI-полировка может незаметно превратить задачу README в
  product redesign. Этого нужно избегать.
- Скриншоты нельзя строить на нестабильных внешних интеграциях.
- Showcase-примеры должны оставаться частью понятной API-истории проекта, а не
  декоративными mock-экранами.

## Критерии готовности

- README выглядит как витрина проекта и объясняет сценарии использования без
  необходимости читать код первым.
- Есть отдельные showcase-примеры для form, chat и adapters.
- Есть воспроизводимый набор screenshot assets.
- Встроенный UI выглядит убедительно в form/chat/output состояниях.
- `make check` проходит.
